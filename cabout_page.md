# Claude Code 修改規格｜Brynn 作品集新增「關於我」獨立頁

請依照以下規格，在現有作品集網站中新增獨立的「關於我」頁面，並同步縮減首頁既有 About 區塊，避免首頁 About、首頁 How I Work 與獨立 About 頁重複。

---

# 0. 本次修改目標

新增一個獨立 About Page，讓招募者能快速理解：

1. Brynn 的職涯背景如何形成現在的 UI/UX 能力。
2. 實際參與與負責過哪些工作，不只看到技能標籤。
3. 加入團隊後，會如何整理需求、確認限制與推進協作。
4. 基本經歷、學歷、獎項與履歷入口。
5. 個人面向只作為輕量補充，不搶走產品設計主線。

此頁**不是第二份線上履歷**，也**不是另一個 How I Work**。

---

# 1. 三個區塊的分工

## 首頁 About

只負責：

- 用一句設計判準建立印象。
- 用一小段背景摘要引導進入 About Page。
- 不展開工作經歷、震時秒判案例、工具與獎項。

## 首頁 How I Work

只負責：

- 呈現 Brynn 的三個設計判斷原則：
  1. 證據先於假設
  2. 每個決策都說得出取捨
  3. 測試必須改變設計
- AI 工作流作為補充。
- 不放職涯時間軸、學歷或工作責任範圍。

## 獨立 About Page

只負責：

- 職涯路徑
- 實際工作範圍
- 團隊合作方式
- Experience / Education / Recognition
- 個人面向與聯絡入口

# 4. 01｜About Hero

## 區塊目的

快速回答：

- 這是誰？
- 有什麼背景？
- 實際工作範圍到哪裡？
- 如何聯絡與查看履歷？

## 桌面版版型

建議 12-column grid：

- 左側：7 欄
- 右側：5 欄

左側放文字，右側放本人照片與基本資訊。

## 頁面標籤

```text
01｜關於我
```

## 主標題

```text
我是 Brynn，
一名 UI/UX 設計師。
```

不使用：

- 產品導向
- 端到端產品設計師
- 研究見長
- 視覺互動見長
- 資深
- Senior

## Hero 內文

請使用以下文案：

```text
我的背景橫跨視覺傳達設計、品牌零售與 UX 顧問專案，工作經驗涵蓋政府數位服務、App 與網站優化。

從使用者研究、資訊架構與 User Flow，到 Prototype、介面設計與規格整理，我會將模糊需求整理成團隊能確認、測試並推進開發的設計內容。
```

## 右側基本資訊

```text
Brynn Li
UI/UX Designer
Taipei, Taiwan
```

CTA：

```text
Email ↗
下載履歷 ↗
```

## 照片規格

- 使用自然、專業的半身照。
- 建議 4:5 或 3:4。
- 背景簡單。
- 不使用證件照風格。
- 不使用 AI 生成人像。
- 若照片尚未提供，使用中性 placeholder。
- placeholder 不可放假人像，可使用純色、姓名縮寫或「Photo」標記。

## About Hero 不放

- 完整年資
- 工作公司時間軸
- 金點獎項故事
- 工具清單
- 設計理念長文
- 「我希望加入……」
- 「目前任職於……」

# 5. 02｜我的職涯路徑

## 區塊目的

解釋三段經歷如何形成現在的 UI/UX 能力，而不是把背景寫成互不相關的履歷。

## 標題

```text
02｜我的職涯路徑
```

## 區塊前言

```text
我的職涯不是從視覺直接跳到 UX，而是逐步把三件事連在一起：資訊如何被看見、人如何做決定，以及數位服務如何被完成。
```

## 版型

桌面版建議三欄卡片或橫向時間軸。

順序：

```text
視覺傳達設計
→ 品牌專櫃銷售
→ UX 顧問專案
```

手機版改為直向排列。

---

## 2.1 視覺傳達設計

標題：

```text
視覺傳達設計
```

文案：

```text
建立資訊層級、版面、視覺敘事與細節控制的基礎，也讓我習慣檢查重要訊息是否能被快速看見、理解，並形成清楚的閱讀順序。
```

此欄傳達：

- 資訊層級
- Layout
- Typography
- 視覺敘事
- UI 細節基礎

不要寫：

- 培養美感
- 對美有高度敏感
- 具備藝術天分

---

## 2.2 品牌專櫃銷售｜4 年

標題：

```text
品牌專櫃銷售｜4 年
```

文案：

```text
長期直接面對顧客如何描述需求、比較選項、提出疑慮並做出決定。我也看見，人們很少在資訊完整、情緒穩定的狀態下做選擇，因此會特別在意資訊是否足以支持判斷與下一步行動。
```

此欄傳達：

- 真實決策場景
- 資訊不足
- 選項比較
- 猶豫與異議
- 下一步行動

不要寫：

- 培養高度同理心
- 具備良好溝通能力
- 提升顧客滿意度與忠誠度
- 未提供數據的成交成果

---

## 2.3 UX 顧問專案

標題：

```text
UX 顧問專案
```

文案：

```text
將前述對資訊與決策的觀察帶入政府數位服務、App 與網站優化。專案經常同時涉及模糊目標、多方需求、既有系統與技術限制，我負責將研究發現整理成資訊架構、流程、原型與介面規格。
```

此欄傳達：

- 真實專案複雜度
- 模糊目標
- 多方需求
- 既有系統
- 技術限制
- Research to Design

## 標題

03｜我實際負責什麼

## 版型

桌面版三欄。

---

## 3.1 Research & Synthesis｜研究與整理

文案：

```text
規劃或執行使用者訪談、易用性測試、認知走查與既有資料整理，歸納使用情境、理解落差、操作問題及設計調整方向。

研究結果不只整理成報告，也會進一步連到後續的流程、資訊層級與介面決策。
```

可顯示 tags：

```text
User Interview
Usability Testing
Cognitive Walkthrough
Research Synthesis
```

Tags 為輔助，不可取代正文。

---

## 3.2 Product Structure｜結構與原型

文案：

```text
將需求與研究發現整理為資訊架構、User Flow、Wireframe 與 Prototype，讓不同方案可以被比較，並在進入開發前提早檢查流程缺口、資訊順序與主要操作。
```

可顯示 tags：

```text
Information Architecture
User Flow
Wireframe
Prototype
```

---

## 3.3 UI & Delivery｜介面與交付

文案：

```text
整理畫面狀態、互動規則、資料條件、元件使用方式與介面規格，並與工程師釐清實作限制，降低設計稿與實際開發之間的理解落差。
```

可顯示 tags：

```text
UI Design
Interaction States
Components
Design Specification
Engineering Collaboration
```

## 此區限制

不要寫：

- 熟悉完整 Design System 建置與治理
- Production-ready code
- 完整前端開發
- 與 PM 協作
- 主導正式產品上線
- 提升開發效率 X%

# 7. 04｜我如何和團隊合作

## 區塊目的

讓 Hiring Manager 能具體想像 Brynn 加入團隊後的工作行為。

## 標題

```text
04｜我如何和團隊合作
```

## 前言

```text
我不把協作理解成增加會議，而是降低資訊落差，讓每一輪討論都有清楚的下一步。
```

## 版型

建議四個橫向 row，或 2 × 2 卡片。

不要使用 icon 取代標題。

---

## 4.1 把口頭需求變成可確認的內容

文案：

```text
面對尚未收斂的討論，我會將內容整理成 User Flow、畫面狀態、方案差異或待確認問題，讓工程師、機關窗口與專案成員能針對同一份內容繼續討論。
```

---

## 4.2 提早把限制帶進設計

文案：

```text
在設計過程中確認資料來源、更新方式、權限、例外情境與技術限制，避免等到設計完成後，才發現流程或介面無法依原方案實作。
```

---

## 4.3 分清楚證據、判斷與假設

文案：

```text
在提案與設計說明中，區分哪些結論來自使用者證據、哪些是基於專案限制的設計判斷，以及哪些內容仍需要驗證，避免將個人偏好包裝成使用者需求。
```

---

## 4.4 保留未決問題與取捨

文案：

```text
當方案無法同時滿足所有條件時，我會說明不同選項各自換到什麼、犧牲什麼，並保留尚未確認的問題，讓團隊清楚知道下一輪需要決定或驗證的內容。
```

## 此區限制

不要寫：

- 擅長溝通
- 善於跨部門合作
- 高度同理
- 能快速配合團隊
- 情緒穩定
- 主動積極

全部以可觀察行為呈現。

## 整體

- 延續現有首頁的米白、黑、棕色與 Serif / Sans 組合。
- About Page 不要改成參考範例的淡紫色履歷模板。

## 區塊間距

桌面：

```text

```

```text
section padding-block: 72–96px
```

依現有 design token 調整。

## 文字行長

中文正文：

```text
max-width: 34–38em
```

避免正文橫跨整頁。

## 標題

- H1：About Hero
- H2：每一大區
- H3：卡片或子項目
- 不跳級
- Section index 可使用 01、02、03

# 14. Accessibility

1. 本人照片需有 alt：
   ```text
   Brynn Li 的個人照片
   ```
2. 純裝飾圖形使用空 alt。
3. 所有 CTA 可鍵盤操作。
4. hover 與 focus 狀態皆需存在。
5. 顏色對比符合 WCAG AA。
6. 使用正確 heading hierarchy。
7. Email 複製按鈕需有成功提示。
8. 外部連結需有可辨識的外連提示。
9. 支援 `prefers-reduced-motion`。
10. 不使用動畫隱藏重要文字。

# 15. Content Safety 與保密

不得新增或顯示：

- 受保密協議限制的專案名稱
- 客戶名稱
- 機關名稱
- 內部畫面
- 未公開功能
- 真實研究資料
- 受訪者資訊
- 內部技術架構
